## 冻结定义

### 1. 客户数据

只放事实层：

- 客户摘要
- 跟进记录

只回答：

- 这个客户当前是什么状态
- 发生过哪些事实记录

不放：

- 问题复盘过程
- 原因猜测
- 方案建议
- 验证结论

### 2. 问题复盘库

独立于客户数据存在，负责存：

- 问题标题
- 发现现象
- 当前问题定义
- 状态门
- 原因猜测
- 方案建议
- 验证结果
- 是否已解决
- 是否已改判

补充字段建议：

- `wrong_judgment_note`
- `redirect_to_problem_id`

### 3. 关联规则

问题复盘库通过关联层连接：

- 关联客户
- 关联销售
- 关联阶段
- 关联时间段
- 关联样本记录

规则：

- 一个问题可关联多个客户样本
- 一个问题可关联一个或多个销售
- 一个问题可关联单个阶段或一个时间段
- 客户数据只作为证据来源，不承担问题闭环状态

### 4. 状态门流转

正常流：

1. 发现问题
2. 原因分析
3. 解决方案
4. 验证中
5. 已解决

特殊流：

1. 待确认问题
2. 已改判
3. 新问题重建

验证结论：

- 有效
- 部分有效
- 无效
- 误判线索

规则：

- 上一步未完成，下一步默认弱化或锁定
- 人工可直接提问题，再找 AI 做原因分析
- AI 也可以先给候选问题，但必须人工确认后才能往下推进
- 如果问题未解决，下次 AI 给建议前必须先检索历史记录和过去失败方案

## 本轮 story

### [ ] STORY-01 三栏结构收口

目标：
把第三工作台固定成 `左选 / 中操 / 右辅`。

验收标准：

- 左栏只负责选择，不承载大段数据
- 中栏最宽，主数据能完整展示
- 右栏固定存在，但不抢主操作空间

### [ ] STORY-02 左栏选择区收口

目标：
左栏固定放主线切换和当前对象列表。

验收标准：

- 有 `市场漏斗复盘 / 销售阶段复盘` 切换
- 有当前对象列表或当前问题列表
- 左栏保持窄，不做大卡片

### [ ] STORY-03 中栏市场漏斗主操作区收口

目标：
市场漏斗视图既要清楚，又不能大到把下面的市场异常提示挤掉。

验收标准：

- 拨号、接通、微信同意、保留入库可完整看清
- 转化率表达清楚
- 市场异常提示尽量一屏能露出

### [ ] STORY-04 中栏销售阶段标准矩阵收口

目标：
把销售阶段视图改成标准表格矩阵，不再用大卡片堆叠。

验收标准：

- 第一列为销售姓名
- 五个阶段列齐全
- 四个转化率列齐全
- 列头、数字、百分比严格对齐
- 表格下方单独放销售阶段异常提示

### [ ] STORY-05 右栏问题闭环辅助区收口

目标：
右栏顺序固定为：

1. 发现问题
2. 原因分析
3. 样本客户链接
4. 建议调整
5. 待验证
6. 已尝试动作折叠区
7. AI skill 选择按钮

验收标准：

- 顺序不乱
- 已尝试动作只在底部折叠区出现
- 样本客户链接位于原因和建议之间，作为证据承接

### [ ] STORY-06 状态门与已改判状态收口

目标：
把状态门做成界面结构的一部分，并明确 `已改判` 不是 `未解决`。

验收标准：

- 上一步未完成时下一步弱化或锁定
- 界面上能看出 `已改判`
- 能看出已改判会指向新问题，而不是覆盖删除旧问题

### [ ] STORY-07 AI 接入口与占位文案统一

目标：
把 `根因猜测`、`建议调整`、`待验证` 三块统一加 AI 标记，并预留 skill 选择按钮。

验收标准：

- AI 标记存在且位置统一
- skill 按钮存在但不接真实逻辑
- 所有 AI 能力都明确还是占位阶段
